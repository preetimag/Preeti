Azure Resource Group

It is a logical container for grouping new or existing resources which will be associated when creating any resource.Example a website with a front end resources for users and database resources to store data. By grouping resources this way simplifies implementation and management by ensuring a resource exist in only one group.However, it can be linked in the portal. Example a SQL database shared between multiple resource groups.

App Service Plans

It allows the user to define based on the region and one of five pricing tiers explained below, which can be shared across app service types.

  1. Free The free tier has limited compute and network bandwidth because the resources are shared and it is intended for evaluation purposes.
  2. Shared The shared tier also shares resources but has increased compute and network bandwitdth intended for low traffic sites.
  3. BasicThe basic tier has dedicated vitual machine with capacity to scale by increasing number of course and RAM, intended to host Web App with no limits.
  4. Standard The standard tier has all the feautes of the basic and also enables Azure Web App features such as deployment slots, automated backups etc.
  5. PremiumThe premium tier has same features as standard with capacity for increase.

At anytime the pricing tier for your plan can be changed without creating a new one. This gives the option of starting in free tier and upgrading as development requirements expand.

Create an Azure Web App using the Azure portal

The Azure portal has user interface to create and manage resources in your subscription. By following the below steps you will have Web App environment with SQL database for publishing Web Application such as ASP.NET MVC application.

Create a Web App and SQL Database

  • Click +new button after signing in to the portal.
  • Select Web+ mobile option in create blade and click Azure Marketplace
  • Select Web App+SQL option to open a blade describing the resources that will be created. Click Create button at the end.
  • Specify the Azure Resource group name and select Web App(Configure required settings) to open Web App blade

Configure require Web App settings

  • Specify a globally unique URL created in *.AzureWebsites.domain for the Web App.
  • Choose an existing App service plan or create a new one using the pricing tier and location options by clicking Viwe All link.
  • Apply the settings and click OK button.

Configure required SQL database settings

  • Select Database(Configure required settings) option and select existing database if u have one or select new database.
  • If new dattabase is selected choose pricing tiers and select Server(Configure required settings).
  • Specify the name for the server and provide admin credentials and apply the server and database settings by clicking the OK button.
  • Click the create button whcih will be enabled after following the above steps.

Add an Azure redis cache to the Azure Resource Group

  • You can add an Azure redis cache by clicking the Add button to provide levels of performance and features in easy way to use.
  • In new resource blade select redis cache option and click Create button to begin configuring the resource
  • Enter a globally unique DNS name and choose pricing and location
  • After adding the redis cache clcik the resource group icon to see all the resouces including redis cache instead of the summary part of the resource group

Create a web app by using ASP.NET web application template

  • From visual studio menu select file > new > project > templates >ASP.NET web application.
  • The authentication defaults to individual user accounts but you can change by clicking change authentication button to he following options.
  1. No authentication do not require users to sign in.
  2. Invdividual user aaccounts store user profiles in a SQL database.

Create a web app using azure resource group template

  • From visual studio menu select file > new > project > templates > azure resource group template.
  • Select Web App+SQL and then Database(Configure required settings) option and select existing database if u have one or select new database.
  • The contents of the deployment project are grouped as follows.
  1. Scripts This folder contains powershell scripts that is used to fully automate creating the web app.
  2. Templates This folder contains two JSON files, the WebSiteSQLDatabase.json file and the WebSiteSQLDatabase.param.dev.json file
  3. Tool This folder contains a copy of AzCopy.exe and is used to copy files to an Azure Storage Account used during deployment.

Connection strings and application strings

It was common practice to store application settings and connection strings in the web.config file for an application, by encripting sensitive information portions of the file. However, sensitive information checked into your source control system that the team has access.


Azure Web Apps offers a better solution whereby your application settings and connection strings can be stored in the web app and then retrieved by your application at runtime by using Role Based Access Control (RBAC), you can restrict access to the production environment.

Set connection strings and app settings in the environment

  • Right-click the web app and select the option to View Settings. Edit the configuration for the web app by clicking the Configuration tab.
  • Scroll down to the Connection Strings section to see the connection string for the SQL database.
  • There are four database types in database type column.
  1. SQL Azure / SQL Database Indicates the connection string is for a SQL database
  2. SQL Server Indicates the connection string is for a SQL server such as an Azure Virtual Machine (or cluster) running SQL Server.
  3. MySQL Indicates the connection string is for a MySQL database.
  4. Custom A connection string to essentially any storage resource.

Retrieve connection strings and app settings from the environment

In your application code, you can retrieve the connection strings and app settings by using the ConfigurationManager class. If you are running your application locally, it will retrieve the values for connection strings and app settings from your configuration files. If you are running the application in Azure, it will retrieve the values for connection strings and app settings from the environment.It is important to note that this strategy works only if you use the same names for connection strings and app settings in your configuration files as you do in your web app environment. When you do, the Azure web app environment setting will take precedence over a setting of the same name in a configuration file. If the setting is not defined in the web app environment, then ConfigurationManager will return the value from the configuration file if it has been set.

Add a deployment slot for an Azure Web App

A deployment slot is an independent web app with its own content, configuration, and even a unique host name and it functions just like any other web app.

Scale to a Standard App Service Plan

  • Deployment slots are available in Standard with up to 5 deployment slots, and Premium provides up to 20, pricing tiers
  • To change the pricing tier, click Browse > Web Apps and click the Pricing Tier
  • In the Pricing Tier blade, select the S1 Standard pricing tier and then click the Select button to apply the change.
  • In the Web App blade, scroll down to the Deployment section and click the Deployment slots to add a deployment slot.

Role Based Access Control

Role Based Access Control (RBAC) is an access control feature in the Azure platform that enables you to manage access to resources in your Azure subscription. With RBAC, you are able to get granular with your access permissions so that only users (or groups) have access to the resources they need.RBAC can be managed at the subscription level by using three built-in roles provided by the Azure platform:

  • Owner Can create and manage resources of all types, including managing access for child resources.
  • Contributor Can create and manage resources but cannot add or delete role assignments for users.
  • ReaderCan read resources but cannot read secrets. For Azure Web Apps, secrets would be things such as application settings, connection strings, and deployment credentials.
